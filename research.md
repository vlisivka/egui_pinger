# Дослідження: Режим «Зупинено» (Stopped) для окремих хостів

## Мета
Додати можливість повністю зупиняти моніторинг (пінг) для окремих хостів, щоб не засмічувати інтерфейс і статистику неактуальними даними. При зупинці всі статистичні дані та історія мають очищатися.

## Технічний аналіз

1. **Зміни у Моделі (`HostInfo`)**:
   - Додавання поля `is_stopped: bool` із властивістю `#[serde(default)]`.

2. **UI (`app.rs`)**:
   - **Колір тексту:** Якщо зупинено, назва та адреса стають сірими.
   - **Текст статусу:** Замість затримки виводиться `tr!("STOPPED")`.
   - **Метрики:** Усі додаткові метрики приховуються.
   - **Керування:** Кнопка `⏹/▶` після кнопки журналу `📋`.
   - **Очищення:** При натисканні ⏹ викликається `reset_statistics()`, що обнуляє все, включаючи `events` (внутрішній журнал).

3. **Логіка (`logic/pinger.rs`)**:
   - **Фільтрація:** Зупинені хости ігноруються в циклах Ping та Traceroute.
   - **Failure deduction:** Такі хости не рахуються як DOWN і не впливають на розрахунок точок відмови.

## Висновки (після рішення користувача)
- Замість «паузи» використовуємо термін **«Зупинено» (Stopped)**, бо це точніше описує скидання статистики.
- Робимо **повне очищення** статистики та журналу при зупинці.
- Додаємо записи-маркери у файл журналу та внутрішній список подій при старті/зупинці.

# Дослідження: Інструмент системної діагностики (System Tools)

## Мета
Створити інтегрований у програму інструмент для запуску стандартних мережевих команд (`ip addr`, `ipconfig` тощо) безпосередньо з графічного інтерфейсу. Це дозволить користувачеві отримувати технічну інформацію про мережеві налаштування системи без потреби відкривати термінал.

## Варіанти реалізації

### 1. Статичний набір команд (Рекомендовано)
Ми визначаємо набір найбільш корисних команд для кожної ОС. Користувач обирає команду з випадаючого списку (ComboBox), бачить її опис і натискає «Запустити».

**Переваги:**
- Безпека (користувач не може виконати довільний код).
- Зручність (не потрібно пам'ятати синтаксис).
- Кросплатформеність (програма сама підставляє правильну команду для поточної ОС).

**Список команд:**
- **Linux:**
    - `Мережеві адреси`: `ip -c addr`
    - `Таблиця маршрутизації`: `ip route`
    - `Активні з'єднання`: `ss -tuln`
    - `Налаштування DNS`: `resolvectl status` (або перегляд `/etc/resolv.conf`)
- **Windows:**
    - `Налаштування мережі`: `ipconfig /all`
    - `Таблиця маршрутизації`: `route print`
    - `Активні з'єднання`: `netstat -an`
    - `ARP таблиця`: `arp -a`

### 2. Вільне введення команд
Текстове поле, куди користувач може вписати будь-яку команду.

**Недоліки:**
- Ризики безпеки.
- Складність для пересічного користувача.
- Потребує додаткової обробки оточення (env).

## Технічний аналіз

1. **Архітектура даних**:
   Створити структуру `SystemCommand`:
   ```rust
   struct SystemCommand {
       label: String,
       description: String,
       cmd: String,
       args: Vec<String>,
   }
   ```

2. **Інтерфейс користувача (UI)**:
   - Нове вікно `SystemToolsWindow`.
   - `egui::ComboBox` для вибору команди.
   - Мітка з описом (Description) під комбобоксом.
   - Кнопка «Запустити» (Run).
   - Поле `TextEdit` (multiline, read-only) для виводу `stdout` та `stderr`.

3. **Виконання (Logic)**:
   - Використання `tokio::process::Command` для асинхронного запуску.
   - Додавання тайм-ауту (наприклад, 10-20 секунд), щоб уникнути зависання при складних командах.

## Вплив на проєкт
- **`src/app.rs`**: Додавання стану вікна та реалізація UI-компонентів.
- **`src/logic/`**: Можливо, створення нового модуля `mod system_commands.rs` для логіки виконання.
- **`locales/uk.po`**: Додавання текстів перекладу для назв та описів команд.

## Рекомендації
Рекомендується реалізувати варіант №1 (статичний набір команд) як найбільш безпечний та дружній до користувача. Кнопку виклику вікна варто розмістити у верхній панелі керування або поруч із налаштуваннями.
