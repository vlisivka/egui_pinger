# Дослідження: Застарілі дані хостів на трасі та неправильна діагностика точки обриву

**Тема**: Аналіз проблем у коді `pinger.rs`, які призводять до того, що після обриву зв'язку деякі хости на трасі тривалий час не оновлюються, що спричиняє неправильне визначення точки обриву.

## Мета

З'ясувати, чому після відключення мережі (напр. WiFi) статуси хостів-хопів на трасі оновлюються повільно або не оновлюються взагалі, і чому це призводить до невірних діагностичних висновків (напр. "ОБРИВ НА 8.8.8.8" замість "ОБРИВ НА Локальний інтерфейс").

## Виявлені проблеми

### Проблема 1: Гонка між визначенням адрес для пінгу та оновленням діагностичного режиму (КРИТИЧНА)

**Файл**: `src/logic/pinger.rs`, рядки 113-197

**Суть**: У головному циклі `pinger_task` крок 2 (визначення `diagnostic_mode` для хопів) виконується **до** кроку 3 (перевірка яких хостів пінгувати). Однак крок 2 перевіряє `status.alive`, яке могло ще **не оновитись** від попередньої ітерації (бо пінг відправляється асинхронно через `tokio::spawn` у кроці 5).

**Хронологія проблеми**:
1. Ітерація N: цільовий хост ще має `alive = true` (стара відповідь)
2. Крок 2: `diagnostic_mode = false` для всіх хопів (бо ціль "жива")
3. Крок 3: хопи **не плануються** для швидкого пінгу (вони у режимі `Slow` = 60с)
4. Крок 5: асинхронний пінг завершується, `alive = false`
5. Ітерація N+1: тепер `diagnostic_mode = true`, але хопи вже мають `next_ping` через 60с
6. Крок 3 (виправлення через `now + interval < *next`): скорочує інтервал, **але лише якщо хоп вже є у `statuses`** і обробляється цією ітерацією

**Наслідок**: Хопи починають пінгуватись швидко із затримкою 1-2 ітерації (100-200мс), що є прийнятно, але **лише якщо вони вже мають запис у `statuses`**. Якщо трасування ще не завершилось — хопи взагалі невідомі.

### Проблема 2: Пінги відправляються послідовно, а не паралельно (КРИТИЧНА)

**Файл**: `src/logic/pinger.rs`, рядки 268-343

**Суть**: Цикл `for (address, _mode, host_info) in addresses_to_ping` виконує для кожного хоста:
1. Перевірку/створення `IcmpEchoRequestor` (включно з DNS-розв'язкою `tokio::net::lookup_host`)
2. `tokio::spawn` для відправки пінгу

Проблема в тому, що **DNS-розв'язка та створення ICMP-запитника виконуються послідовно** в основному циклі. Якщо мережа відключена, `tokio::net::lookup_host` може **блокувати** на системний DNS-таймаут (зазвичай 5-30 секунд!) для кожного хоста, який не є IPv4/IPv6-адресою.

**Наслідок**: Якщо є 5 хостів з доменними іменами, основний цикл може зависнути на 25-150 секунд, протягом яких **жоден хоп не оновлюється**.

### Проблема 3: Кеш `requestors` не інвалідується при зміні мережі (ПОМІРНА)

**Файл**: `src/logic/pinger.rs`, рядки 49, 273-309

**Суть**: `requestors: HashMap<String, IcmpEchoRequestor>` створюється один раз і **ніколи не очищується**. Якщо мережеві інтерфейси змінились (напр. WiFi вимкнувся і увімкнувся знову з іншою IP-адресою), старі `IcmpEchoRequestor` можуть бути прив'язані до неіснуючого сокета.

**Наслідок**: Після перевмикання WiFi, пінги можуть постійно фейлитись через застарілі ICMP-сокети, доки програму не перезапустять. Навіть якщо мережа відновилась, `requestor.send()` може повертати `Err`, і створення нового requestor'а не відбувається — бо старий запис вже є у кеші.

### Проблема 4: Логіка визначення "Local Interface" не враховує стан першого хопа (КРИТИЧНА)

**Файл**: `src/logic/pinger.rs`, рядки 224-255

**Суть**: Поточна логіка визначає обрив як "Local Interface" лише якщо:
- `path.is_empty()` — трасу ще не виявлено
- `path.len() == 1 && path[0] == target_addr` — траса містить лише ціль

Але **не перевіряється** типовий сценарій локального обриву: коли **перший хоп (шлюз)** також недоступний. Якщо WiFi вимкнено, то **всі** хопи на шляху будуть "broken", і алгоритм просто знайде перший з них як failure_point. Але перший хоп на трасі — це зазвичай маршрутизатор (192.168.1.1), а не "локальний інтерфейс".

**Правильна логіка**: Якщо перший хоп на трасі (шлюз) недоступний, це з великою ймовірністю означає локальний обрив мережі. Потрібно додати перевірку: якщо `hop_idx == 0` і хоп є broken, то це "Local Interface" (або "Default Gateway").

### Проблема 5: Failure deduction виконується на старих даних (КРИТИЧНА)

**Файл**: `src/logic/pinger.rs`, рядки 199-266

**Суть**: Крок 4 (failure deduction) виконується **ПІСЛЯ** кроку 3 (визначення хостів для пінгу), але **ДО** відправки пінгів (крок 5). Це означає, що failure deduction працює на даних **з попередньої ітерації**. Але ще важливіше — він працює на даних хопів, які можуть бути **дуже старими**.

Хопи в режимі `Slow` (60с) оновлюються раз на хвилину. Якщо обрив стався щойно, failure deduction побачить для хопів дані хвилинної давності. Навіть після активації `diagnostic_mode`, потрібно чекати **мінімум 3 пінги × 2с = 6с** щоб `streak >= 3` стало істинним для хопів.

**Наслідок**: Протягом перших ~60 секунд після обриву, деякі хопи ще мають `streak_success = true` або `streak < 3`, що спотворює визначення точки обриву.

### Проблема 6: Відсутність таймштампу "свіжості" даних хоста (ПОМІРНА)

**Файл**: `src/model/status.rs`

**Суть**: Структура `HostStatus` не має поля `last_updated: Instant`. Failure deduction не може відрізнити "хоп відповів успішно 5 секунд тому" від "хоп відповів успішно 55 хвилин тому". Обидва мають `streak_success = true, streak = N`.

**Наслідок**: Якщо хоп у режимі `Slow` відповів 50 хв. тому, і зараз мережа впала, failure deduction вважає цей хоп "живим" ще 10 хвилин, поки не прийде наступний пінг.

### Проблема 7: Хопи без статусу вважаються "робочими" у failure deduction (ПОМІРНА)

**Файл**: `src/logic/pinger.rs`, рядки 239-246

**Суть**: У перевірці `all_further_broken`:

```rust
for next_hop in &path[hop_idx + 1..] {
    if let Some(nh_status) = state_lock.statuses.get(next_hop) {
        if nh_status.streak_success || nh_status.streak < 3 {
            all_further_broken = false;
            break;
        }
    }
    // Якщо statuses.get(next_hop) повернув None — хоп ігнорується!
}
```

Якщо хоп ще не має запису в `statuses` (бо не встиг створитись чи бо traceroute ще не завершився повністю), він **ігнорується**, а не вважається "невідомим". Це може призвести до хибно-позитивного `all_further_broken = true`.

### Проблема 8: Traceroute виконується лише кожну годину (ПОМІРНА)

**Файл**: `src/logic/pinger.rs`, рядки 57-80

**Суть**: `last_trace_times` перевіряється на `> 3600 секунд`. Це означає, що після того як маршрут виявлено один раз, він **не перевідкривається** протягом 1 години. Але при зміні мережі (WiFi → мобільна, VPN on/off) маршрут може повністю змінитись.

**Наслідок**: Якщо мережа змінилась, failure deduction працює на **застарілому маршруті**, перевіряючи хопи, через які трафік вже не йде.

### Проблема 9: `streak` та `streak_success` ініціалізуються неправильно (ПОМІРНА)

**Файл**: `src/model/status.rs`, рядки 176-180

**Суть**: `HostStatus::default()` встановлює `streak = 0` і `streak_success = false`. Це означає, що новостворений хоп без жодного пінгу має `!streak_success && streak >= 3` = `true && false` = **false**. Це саме по собі коректно (streak < 3), але разом з проблемою 5 означає, що новий хоп потребує мінімум 3 невдалих пінги, щоб failure deduction його врахував.

При Fast mode (2с) це = 6 секунд. Але при Slow mode (60с) хоп без diagnostic_mode почне пінгуватись через 60с, потім ще 3×60с = 180с = **3 хвилини** мінімум до першого визначення failure_point.

### Проблема 10: `alive` поле встановлюється незалежно від `add_sample` (НЕЗНАЧНА)

**Файл**: `src/logic/pinger.rs`, рядки 329-331

**Суть**: `status.alive` і `status.add_sample(rtt_ms)` — це два окремих оновлення, які семантично мають бути одним. `add_sample` оновлює `streak`, `streak_success`, `latency` тощо, але `alive` встановлюється окремо до виклику `add_sample`. Це не створює гонку (бо mutex), але з логічної точки зору краще встановлювати `alive` всередині `add_sample`.

## Рекомендації

Рекомендую виправити проблеми в такому порядку (від найвпливовіших):

### 1. Додати таймштамп свіжості до `HostStatus` (вирішує проблеми 5, 6)

Додати поле `last_updated: Option<Instant>` до `HostStatus`. У failure deduction **ігнорувати дані, яким більше ніж N секунд** (напр. 2× інтервал Fast mode = 4с чи навіть чогось більшого, як 15с). Якщо дані хопа старі — не вважати його "живим".

```rust
let data_is_fresh = h_status.last_updated
    .map(|t| now.duration_since(t) < Duration::from_secs(15))
    .unwrap_or(false);
let hop_is_broken = !h_status.streak_success && h_status.streak >= 3;
let hop_is_stale = !data_is_fresh; // дані застарілі — вважаємо зламаним
```

### 2. Покращити логіку визначення "Local Interface" (вирішує проблему 4)

Додати перевірку: якщо перший хоп трасу також broken, це "Local Interface" / "Default Gateway" обрив:

```rust
if path.is_empty() {
    found_point = Some("Local Interface".to_string());
} else {
    // Якщо перший хоп (шлюз) broken — це локальний обрив
    if let Some(first_hop_status) = state_lock.statuses.get(&path[0]) {
        if !first_hop_status.streak_success && first_hop_status.streak >= 3 {
            found_point = Some("Local Interface".to_string());
        }
    }
    // ... інша логіка для несуміжних пошкоджень
}
```

### 3. Паралелізувати DNS-розв'язку та створення requestor'ів (вирішує проблему 2)

Замість послідовного циклу `for address in addresses_to_ping`, використати `futures::future::join_all` або `tokio::JoinSet` для паралельної обробки всіх хостів. Або винести DNS-розв'язку в окремий cache з TTL.

### 4. Інвалідувати кеш requestor'ів при мережевих помилках (вирішує проблему 3)

Якщо `requestor.send()` повертає `Err`, видалити його з кешу:

```rust
// Всередині tokio::spawn:
Err(_) => {
    // Видалити застарілий requestor
    // (потрібно передати Arc<Mutex<HashMap>> замість клонування state)
    (false, f64::NAN)
}
```

### 5. Перезапускати traceroute при зміні мережі (вирішує проблему 8)

Додати механізм детекції зміни мережі: якщо кілька цільових хостів одночасно перейшли зі стану "alive" в "down" — скинути `last_trace_times` і перезапустити traceroute.

### 6. Вважати хопи без статусу "невідомими" (вирішує проблему 7)

```rust
if let Some(nh_status) = state_lock.statuses.get(next_hop) {
    if nh_status.streak_success || nh_status.streak < 3 {
        all_further_broken = false;
        break;
    }
} else {
    // Хоп без статусу — невідомий, не вважати "broken"
    all_further_broken = false;
    break;
}
```
