# Дослідження: Типізований журнал подій (100k) з локалізацією та фільтрацією

**Тема**: Розробка високоефективного логера для `egui_pinger` з підтримкою 100 000 подій, мовою користувача та детальним відстеженням станів мережі.

## Мета

Реалізація архітектури **Варіанту Б**, яка забезпечує:
1. Зберігання до **100 000 подій** у `VecDeque<LogEntry>` (економія пам'яті).
2. **Локалізацію**: всі записи в журналі та файлі генеруються мовою інтерфейсу (через `tr!`).
3. **Повну історію**: включення таймаутів (timeouts) та подій відновлення зв'язку.
4. **Фільтрацію**: вибірковий перегляд типів подій у GUI.

## Модель даних (LogEntry)

```rust
// src/model/status.rs

#[derive(Debug, Clone, Serialize, Deserialize)]
pub enum LogEntry {
    /// Успішний пінг або таймаут
    Ping {
        timestamp: u64,
        seq: u32,
        rtt: Option<f32>, // None = Timeout
        bytes: u16,
    },
    /// Періодична статистика
    Statistics {
        timestamp: u64,
        mean: f32,
        median: f32,
        p95: f32,
        jitter: f32,
        mos: f32,
        loss: f32,
        sent: u32,
        lost: u32,
        // та інші поля...
    },
    /// Зміна маршруту
    RouteUpdate {
        timestamp: u64,
        path: Vec<String>,
    },
    /// Інцидент (Втрата/Відновлення)
    Incident {
        timestamp: u64,
        is_break: bool, // true = втрата, false = відновлення
        streak: u32,    // кількість пакетів у поточному стані
        downtime_sec: Option<u64>, 
    },
}
```

## Формат та Локалізація

Журнал генерується динамічно. Для запису у файл та виводу в GUI використовуються однакові правила форматування з підтримкою `tr!`.

### Приклад виводу (якщо обрана українська мова):

| Тип | Формат рядка (приклад) |
|---|---|
| **Пінг** | `[18:40:01] 64 байт від 8.8.8.8: icmp_seq=1001 час=12.5 мс` |
| **Таймаут** | `[18:40:02] Таймаут запиту для icmp_seq=1002` |
| **Статистика** | `# [18:40:03] Статистика: Сер=12.5 Мд=12.4 P95=15.0 Дж=1.2 MOS=4.4 Втрати=0.1%` |
| **Траса** | `% [18:40:05] Маршрут оновлено: 192.168.1.1 -> ... -> 8.8.8.8` |
| **Втрата** | `! [18:41:00] Зв'язок втрачено (3 запити без відповіді поспіль)` |
| **Втрата** | `! [18:41:00] Зв'язок втрачено на 192.168.1.1` |
| **Відновлення** | `! [18:42:02] Зв'язок відновлено (час простою: 62 с)` |

## Деталі реалізації

### 1. Ефективний буфер на 100 000 записів
- Використовуємо `VecDeque` для автоматичного витіснення старих подій під час досягнення ліміту.
- **Пам'ять**: ~2.5 МБ. При 100 000 записах це дозволяє зберігати історію за кілька годин (або навіть днів при повільних режимах) без навантаження на систему.

### 2. GUI з фільтрацією та віртуалізацією
- **Фільтри**: Чекбокси "Пінги", "Таймаути", "Статистика", "Траса/Інциденти".
- **Віртуалізація**: `egui::ScrollArea` з фіксованою висотою рядків для швидкої прокрутки 100к записів.
- **Кольори**: Диференціація за типом (червоний для втрат, зелений для статистики тощо).

### 3. Робота з Лог-файлом
- Рядки пишуться у файл **мовою, яка була активна в момент події**.
- При зміні мови інтерфейсу нові записи в файл будуть писатися новою мовою.
- При старті логування пишеться: `=== Початок журналу: {дата/час} ===`.
- При зупинці: `=== Кінець журналу: {дата/час} ===`.

## Вплив на проєкт

- **`src/logic/pinger.rs`**: Логіка створення об'єктів `LogEntry` та їх додавання у `VecDeque`.
- **`src/model/status.rs`**: Опис структури лога та методів форматування.
- **`src/app.rs`**: Вікно перегляду з фільтрами та механізм ініціалізації типової назви файлу.
- **I18n**: Додавання нових ключів у `.po` файли для перекладу повідомлень журналу.

## Висновок
Обраний варіант дозволяє створити професійний "бортовий журнал", який є одночасно і компактним, і інформативним. Використання `tr!` гарантує, що користувач отримає зрозумілі звіти про стан мережі своєю мовою.
